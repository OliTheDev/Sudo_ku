1.1.1 ID structure

given just an elements id we need to know three things.
  • what box it's in
  • what row it's in
  • what column it's in?

this means each id will consist of three characters that can be extrapolated to reviel the above information.
the first 2 characters will signify which box the cell is located within.
the first character can only be one of three letters a, b and c each representing the row of the containing box.
the second character can only be one of three letters d, e, and f each representing which column the containing box is in.
a visual example is given below as to how the grid is subdivided.
AD AE AF
BD BE BF
CD CE CF

the final character is a number representing the specific cell within that subgrid going left to right top to bottom.
an example would ad1 which represents the top most call on the far left.
so the AD subgrid would look like this

ad1 ad2 ad3
ad4 ad5 ad6
ad7 ad8 ad9

Data types

the game is played on a 9 by 9 grid this mean 81 indavidual cells. Each cell will have a containing value the other piece of information we need to keep track of is the containing number within each cell.
In this case we use 0 to represent an empty cell so the contenets will only ever be a sinlge digit.
the result of this is 81 key value pairs that will be constantly updated.
because of these needs a hashmap is the ideal datastructure.
It is simply named map and placed in the global scope.

the game is played by adding numbers to a grid. some cells in this grid already have number that cannot be changed. when a number is added there are three
requirements it must satify. the number being added my not appear anywhere else in the same row. the number being added my not appear anywhere else in the same column
and lastly the number being added my not appear anywhere else in the same subgrid. the game is finished when all the cells of the grid are filled. there are three layers to every project. data, logic and presentation. I think its important to go through them in tha order. so
